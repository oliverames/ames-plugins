# ames-Codex

Oliver's personal plugin marketplace for Codex with experimental Codex dual-host support. Ships 7 plugins, 46 skills, 15 MCP servers.

## Structure

```
plugins/<name>/                             Plugins (each has .Codex-plugin/plugin.json)
plugins/ames-standalone-skills/skills/      Original skills (28)
plugins/ames-community-skills/skills/       Curated third-party skills (1: humanizer)
plugins/build-ios-apps-codex/skills/        iOS skills converted from OpenAI's Codex plugin (6)
plugins/build-macos-apps-codex/skills/      macOS skills converted from OpenAI's Codex plugin (11)
.Codex-plugin/marketplace.json             Codex marketplace manifest
.agents/plugins/marketplace.json            Codex marketplace manifest (experimental)
plugins/<name>/.codex-plugin/plugin.json    Codex plugin manifest (for dual-host plugins)
sync                                        Regenerates marketplace.json from plugin manifests
bump-and-sync                               Bumps ames-ynab version + runs sync + pushes
```

## Workflow

1. **Develop** — edit plugins and skills directly
2. **Sync** — run `./sync` to regenerate marketplace manifests
3. **Commit and push** — Codex polls the manifest and auto-updates installed plugins

`marketplace.json` is auto-generated by `./sync`; never edit it directly.

## Dual-host support

The repo carries two parallel manifest namespaces under a shared plugin tree:

| Host | Marketplace manifest | Per-plugin manifest | Plugins |
|------|----------------------|---------------------|---------|
| Codex (primary) | `.Codex-plugin/marketplace.json` | `.Codex-plugin/plugin.json` | 7 |
| Codex (experimental) | `.agents/plugins/marketplace.json` | `.codex-plugin/plugin.json` | 5 (excludes `build-ios-apps-codex`, `build-macos-apps-codex`) |

Skill content (`SKILL.md`) and MCP config (`.mcp.json`) are portable across hosts. Only the plugin and marketplace manifests differ. `build-ios-apps-codex` and `build-macos-apps-codex` are Codex-only because their skills were converted from OpenAI's Codex plugins (which exist upstream in `openai/plugins`) and cannot round-trip.

When bumping a plugin version, update all three spots: `plugins/<name>/.Codex-plugin/plugin.json`, `plugins/<name>/.codex-plugin/plugin.json` (if it exists), and the per-plugin `version` in the root `.Codex-plugin/marketplace.json`.

## Plugin conventions

- Every plugin needs `.Codex-plugin/plugin.json` with `name`, `version`, `description`, `author`
- For Codex support, also add `.codex-plugin/plugin.json` with the Codex schema (includes `interface`, `category`, `capabilities` fields)
- `version` is required — bump it whenever content changes so clients detect an update
- MCP plugins declare servers in a flat `.mcp.json` at the plugin root (no outer `mcpServers` wrapper, no `"type": "stdio"`)
- Use `${ENV_VAR}` in `.mcp.json` for secrets — resolved from Codex `settings.json` env at runtime
- Skills go in `skills/<name>/SKILL.md` within the plugin directory
- Commands go in `commands/<name>.md` with frontmatter (`name`, `description`, `allowed-tools`)

## Skill conventions

- Each skill is a folder with a `SKILL.md` inside `plugins/<plugin-name>/skills/`
- SKILL.md frontmatter: `name` and `description` are required
- **`name` MUST be kebab-case matching the directory name** (e.g. `name: apple-notes-formatting` for `skills/apple-notes-formatting/`). Display-name style values like `"Apple Notes Formatting"` cause Cowork's marketplace validator to reject the entire plugin with a generic "Failed to update marketplace" error; see memory `ref_claude_skill_md_schema.md`.
- `./sync` regenerates `marketplace.json` from plugin.json files. Skills are auto-discovered by Codex at install time from the `skills/` directory — do NOT list them explicitly in marketplace.json (causes path resolution issues).
- Every subdirectory of `skills/` MUST contain a `SKILL.md`. Dev workspaces without a SKILL.md belong elsewhere (Cowork's validator trips on them).

## Version bumping

- **Patch** (1.0.0 → 1.0.1): content tweaks, bug fixes
- **Minor** (1.0.0 → 1.1.0): new skills, commands, or MCP servers added; new user-facing capability
- **Major** (1.0.0 → 2.0.0): breaking changes or full rewrites

`bump-and-sync` handles version bumping + sync + push for `ames-ynab` automatically (called by the npm postpublish hook in the ynab-mcp-server repo).

When bumping a plugin under dual-host, keep Codex and Codex plugin.json versions in lockstep. The root marketplace.json entry must also match.

## Marketplace

Registered in Codex as `ames-Codex` pointing to `oliverames/ames-Codex`. After any change: run `./sync`, commit, push. Codex will detect the update on next marketplace refresh.

For Codex (experimental): `codex marketplace add https://github.com/oliverames/ames-Codex`.

## Install paths

| Component | Location |
|-----------|----------|
| Plugin cache | `~/.Codex/plugins/cache/ames-Codex/<name>/<version>/` |
| Marketplace cache | `~/.Codex/plugins/marketplaces/ames-Codex/` |
| Plugin registry | `~/.Codex/plugins/installed_plugins.json` |
| User-level marketplace registration | `~/.Codex/settings.json` `extraKnownMarketplaces` |
| User-level enabled plugins | `~/.Codex/settings.json` `enabledPlugins` |

## Gotchas

- **Plugin name must not equal marketplace name** (`ames-Codex`). Causes install collision.
- **`marketplace.json` is auto-generated**; run `./sync`, never edit manually.
- **MCP connector secrets**: plugin-spawned MCP servers don't inherit the `op run` environment. Secrets must be declared in `.mcp.json` via `${ENV_VAR}` (resolved from `settings.json` env block), or loaded from a plugin-local `.env` file by the server itself.
- **Flat `.mcp.json` format**: no outer `mcpServers` wrapper, no `"type": "stdio"`. Matches the official plugin format.
- **Codex marketplace schema differs from Codex's**: Codex uses nested `source: {source: "local", path: "./..."}` and requires `policy: {installation: "AVAILABLE", authentication: "ON_INSTALL"}`. Codex uses flat `source: "./..."`. Both manifest shapes are maintained side-by-side.
- **`build-ios-apps-codex` and `build-macos-apps-codex` are Codex only** by design. Their skills were converted from OpenAI's Codex plugins and cannot be republished to Codex (they already exist upstream in `openai/plugins` in their original form). Both are promoted to top-level plugins (rather than nested inside an umbrella plugin) so Codex's shallow auto-discovery finds each sub-skill directly.
- **Version parity across manifests** is enforced manually. The `wrap-up` skill's config-drift check catches mismatches at end of session.
